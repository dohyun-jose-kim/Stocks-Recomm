## 4. API 응답 구조 검증

`curl`로 두 플랫폼 직접 두드림.

**Polymarket Gamma API** (`https://gamma-api.polymarket.com/events`)
- 인증 불필요
- 한 events 호출에 events 100개 (limit 무시) — pagination 필요
- **카테고리 필드 없음** → slug/title 키워드 매칭 필요
- 정렬 기본값에서 Fed 관련이 잘 안 나옴 → `order=volume24hr&ascending=false` 명시 필요
- `outcomePrices`, `outcomes`는 문자열 인코딩된 JSON

**Kalshi Trade API** (`https://api.elections.kalshi.com/trade-api/v2/markets`)
- 인증 불필요 (읽기)
- `series_ticker` 파라미터로 깔끔하게 시리즈 필터
- `yes_bid_dollars`, `yes_ask_dollars`, `last_price_dollars` 다 제공
- KXFED 한 시리즈에 109개 마켓 (8개 회의 × 회의당 11~18 threshold)

## 7. 정보화 단계 — derived 테이블

질문: "127개 raw 마켓 → 더 의미있는 단위로 합칠 수 있나? 시계열로도?"

**해결**: `event_summaries` 테이블 추가. event_ref(같은 사건 묶음) 단위로 마켓들을 분포 dict로 합쳐 1행씩 시계열 누적.

결과: 127 raw → **10개 event summaries**

| 형태 | event 수 | 합 (total_prob) |
|---|---|---|
| categorical (mutually exclusive) | Polymarket 2개 | ~1.0 |
| cumulative (threshold) | Kalshi 8개 | 4.0~12.7 |

총합 `total_prob`로 형태를 자동 구분 가능. cumulative → PMF 변환은 explore 단계 작업으로 미룸.

시그널 예:
- Polymarket 6월 FOMC: **동결 97.8%**
- Polymarket 2026 인하 횟수: **0회 69.6%, 1회 17.5%**
- Kalshi 6월 FOMC: 금리 3.50~3.75% 구간 95% 컨센서스

## 8. 스케줄링 결정

질문: "30분 해상도가 정말 필요한가? 컴퓨터를 끄면 안 되나?"

**결정**: GitHub Actions로 클라우드에서 cron 실행. 하루 2회.
